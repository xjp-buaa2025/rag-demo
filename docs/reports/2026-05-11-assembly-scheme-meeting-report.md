# 航空发动机装配方案设计学科智能体 —— 组会汇报

> 日期：2026-05-11
> 主题：从"通用 RAG 工具"到"装配工艺学科智能体"的构想

---

## 1. 一句话定位

把现有的 RAG / 知识图谱 / CAD / BOM 工具，沿着"航空发动机装配方案设计"这条线，组装成**一个能像装配工艺专家一样思考、能产出真实可落地的装配方案、能在使用中持续生长**的学科智能体。

---

## 2. 为什么要做这件事

### 2.1 现状：工具有了，但还不是"学科专家"

我们这一年陆续搭起了一个相当完整的 RAG 系统：

| 能力 | 已实现 |
|------|--------|
| 文本检索 | bge-m3 向量 + BM25 + 重排序 |
| 多模态 | Chinese-CLIP 图文检索 + Vision LLM 描述 |
| 知识图谱 | Neo4j + BOM/手册/CAD 三源融合 + 黄金三元组评测集 |
| 文档解析 | RAGFlow deepdoc + ATA 章节识别 |
| 智能体框架 | LangChain + LangGraph 多步骤管道 |

这套系统**能回答问题**，但**不会做工程**。

打个比方：它像一个博闻强识的图书管理员——你问什么，它能找到对应的章节段落给你看。但你让它"为某型发动机的压气机段设计一份装配方案"，它就不会了，因为这不是检索问题，而是**学科任务**。

### 2.2 缺什么？缺"专家心智"这一层

一个真正的工艺专家拿到这种任务，脑子里有的不是检索能力，而是：

- 一套**结构化的工作流**：需求分析 → 概念设计 → 工艺规程 → 评审
- 一套**领域方法论**：QFD、DFA、公差链、动平衡、选配、预紧力管理…
- 一套**判断准则**：什么时候该问、什么时候该查、什么时候该返工
- 一套**对"好方案"的认知**：什么样的产物是合格的、缺什么字段说明做得不专业

这些东西**不在我们的 RAG 知识库里**——它是"用法"而不是"内容"。这就是"工具 vs 智能体"的真正差距。

### 2.3 选题：为什么是"装配方案设计"

三个理由：

1. **学科深度足够**：装配方案设计本身是一个完整工程任务，涵盖需求→设计→工艺→评审，能撑起一个"学科智能体"的厚度
2. **与现有数据天然闭环**：PT6A 手册 / BOM / CAD / 知识图谱已就位，案例研究材料现成
3. **学术与产业价值兼具**：航空发动机装配是国家战略短板，做出真实可演示的方案设计能力，对论文和实际工业应用都有意义

---

## 3. 借鉴"Skill"的设计哲学

参考了 Anthropic 提出的 **Skill 设计范式**——把一个领域的方法论、工作流、产物模板封装成一个可被智能体加载的"专家心智支架"。

**核心思想**：方法论不应该藏在代码里，而应该作为**显式文档**存在，让人和机器都能读、都能修。代码只负责"按方法论调度工具"，不藏方法论。

这种"代码薄、方法论厚"的范式，让整个系统具备了：

- **可解释**：陛下打开 skill 文件夹就能看到所有方法论与决策依据
- **可演进**：随着用得越多，方法论可以被人和机器共同精炼
- **可迁移**：换个领域（比如涡轮叶片设计、燃烧室设计）做新 skill 即可，不用重写代码

---

## 4. 整体思路：四个核心设计抉择

### 4.1 三层架构：方法论 / 执行 / 交互各司其职

```
┌───────────────────────────────────────────────────────┐
│ 第三层：前端向导（人在回路）                          │
│   5 个阶段卡片 + 双轨 HITL                            │
└───────────────────────────────────────────────────────┘
                          ↑
┌───────────────────────────────────────────────────────┐
│ 第二层：执行层（管道调度）                            │
│   把方法论翻译成工具调用：RAG / KG / CAD / Web / LLM │
└───────────────────────────────────────────────────────┘
                          ↑
┌───────────────────────────────────────────────────────┐
│ 第一层：Skill 文档（专家心智）                        │
│   方法论 + 产物模板 + 决策图 + 黄金范例 + 反模式      │
└───────────────────────────────────────────────────────┘
```

**第一层是灵魂**：所有"专家怎么想"都长在 markdown / JSON 文件里。第二层是肌肉。第三层是接口。

### 4.2 5 阶段工作流：上下游贯通的设计谱系

经过两轮讨论收敛——这个 skill **不只做工艺规程**，而是**从概念设计一路贯通到详细装配方案**：

| 阶段 | 做什么 | 输出 |
|------|--------|------|
| **S1 任务调研** | 把陛下的模糊需求结构化；联网拉参考资料；查本地 KG | 任务说明书（含目标/范围/边界/约束/已知风险五栏）|
| **S2 需求分析** | 量化指标；识别关键尺寸特征（KC）；DFM/DFA 评分 | 需求指标矩阵 + 装配可制造性诊断 |
| **S3 概念设计** | 给出 3-5 个备选架构；做装配仿真预检；统一基准 | 推荐架构 + 模块/接口/优劣对比 |
| **S4 详细工艺** | 工序总表 + 工装规划 + 公差分配 + 关键件特殊控制 | 装配工艺规程（4 子模块） |
| **S5 评审导出** | 三角色虚拟评审 + 追溯性矩阵 + 局部迭代 | 最终方案 Markdown 文档 |

这个流程**同构于教科书里的"学科设计三段论"**，但每个阶段都被锚定到具体的工业实践动作。

### 4.3 双轨 HITL：人和模型各自负责自己擅长的事

**痛点**：传统智能体要么"全自动跑完出错谁都看不出"，要么"每步问用户拖累节奏"。

**设计**：每个阶段都给陛下**两条路径**同时存在：

- **直接编辑**：陛下点字段就能改（适合微调：改个数值、调个工序顺序）
- **自然语言指导**：陛下打"第 5 道工序要在第 3 道之前，因为叶片要先装入"，模型拿着指导意见 + 当前产物**重新生成**

这种设计让陛下**既能精细控制又能大开大合**，符合实际工程师的工作习惯。

### 4.4 运行时反哺：让 Skill 越用越聪明

**痛点**：传统 skill 是静态的，写完就死。

**设计**：每次跑流程时，联网检索到的 GJB 标准、Service Bulletin、最佳实践，**经陛下审核后**双向写入：

- 进 **skill 自己的参考库**（下次跑同子系统时自动加载）
- 进 **RAG 知识库**（之后任何问答都能检索到）

这意味着 skill **每被使用一次就长大一点**——这是它区别于普通工具最关键的一点。

## 5. 真专家视角的反思：教科书味 vs 工业实际

设计第一轮收敛出的 5 阶段还偏"教科书味"。第二轮以"航发装配工艺师"视角重新审视后，明确补上了 5 处工业级关键动作：

| 工业实际需要 | 学科价值 |
|--------------|----------|
| **公差链分析（Tolerance Stack-up）** | 航发装配真正的难点不是"怎么装"，而是装完之后的关键间隙（叶尖/轴向）能不能合规——必须做反向工艺设计 |
| **DFM/DFA 装配可制造性审查** | 零件设计有缺陷不可能装出好方案，要前置审查 |
| **关键件特殊控制** | 转子动平衡、选配/配磨/配研、预紧力管理、关键件追溯性——这些是航发与一般机械装配的本质差异 |
| **装配仿真预检** | DELMIA / Process Simulate 是现代航发标配；用 KG + CAD 至少做可达性分析 |
| **试装迭代** | 评审不是单向的"出报告"，要允许"发现问题→回 S4 局部重做"的循环 |

这些点不写进方法论文档，整个 skill 就只是"会答题的本科生"，写进去就变成"懂工程的工艺师"。

## 6. 示范标的：PT6A 高压压气机（HPC）

选 PT6A HPC 作首版示范标的的理由：

1. **数据闭环**：陛下手上的 P&W Canada 维修手册 72-30 章 + 黄金三元组（108 实体 / 99 三元组）正是这个子系统
2. **复杂度适中**：3 级轴流 + 1 级离心 ≈ 35 零件 / 13 总成，足以覆盖 4-5 类装配难点（叶片装入、动平衡、间隙控制、紧固件扭矩）
3. **论文友好**：与现有 PT6A 实验闭环，可作为论文 case study

未来 skill 范式稳固后，可推广到涡轮段、燃烧室、附件齿轮箱等其他子系统。

---

## 7. 参考资料的"双轨"策略

不闭门造车——skill 内置一份**预烘的黄金参考库**（GJB 5060A、ASME Y14.5/B89.7.5、HB 系列、PT6A 手册、航发设计手册第 19 册等），同时在运行时通过 web 检索补充该具体子系统的最新资料。**通用方法论沉淀在本地，最新具体事实从联网获取**——既不脱节，也不僵化。

---

## 8. 整个工作的预期产出

按重要性排序：

1. **一份可被加载、可被读、可被修的 Skill 文档**——这是学科智能体的"灵魂"
2. **一套覆盖 5 阶段的工作流页面**——让陛下能亲自走完一个完整设计
3. **PT6A HPC 完整装配方案范例**——作为方法论的"实例化"，同时是论文 case study
4. **运行时反哺机制下的"自我生长"过程**——可作为论文里"人机协作"的量化证据
5. **可迁移的范式**——任何子系统、任何学科任务都能照这个模板复制一份新 skill

---

## 9. 学术叙事

这件事不是单纯"做个 demo"，而是回应一个核心问题：

> **当 RAG 工具越做越好时，下一步该怎么把它们组装成"会做工程"的智能体？**

我们的回答是：

- 不要让所有方法论都隐藏在代码里
- 要让一个领域的"专家心智"成为可被加载的显式资产
- 要让人在关键节点能介入但又不被打扰
- 要让系统在使用中长大

**装配方案设计** 是验证这个回答的载体；**PT6A HPC** 是首个实例；**Skill 范式** 是可被推广的方法。
